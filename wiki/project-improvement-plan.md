# 项目改进计划

最后更新：2026-03-24

这份文档用于记录当前项目的工程改进项。目标不是做大重构，而是按“个人项目最高性价比”的顺序，把最容易踩坑、最影响稳定性的环节逐步补齐。

## 已完成

- [x] 修复监听重复转发问题
  说明：解决重复注册消息处理器导致同一条消息被重复转发的问题，并补了短时去重保护。

- [x] 完善发布通知链路
  说明：将 Telegram 通知并入 `release.yml`，避免 `GITHUB_TOKEN` 创建 Release 后无法触发独立通知工作流的问题。

- [x] 完善广告异常回退闭环
  说明：保留广告模块为启动硬依赖，但将远程广告数据异常降级为默认头部、按钮、链接回退，避免消息链路被一并拖垮。

- [x] 打通 Telethon 代理链路
  说明：将数据库配置和环境变量代理真正接入 `TelegramClient(...)`，支持 `socks5`、`http`、`mtproxy` 三种生效路径，并在代理切换后重建客户端。

## 下一阶段

- [ ] 提升转发消息的 Markdown/HTML 稳定性
  目标：避免正文包含 `_ * [ ] ( )` 等字符时，Bot API 解析失败或渲染错乱。

- [ ] 补最小测试集
  目标：先覆盖关键词匹配、黑名单过滤、重复启动保护和消息去重，保证高风险链路改动有回归保护。

- [ ] 收敛启动流程
  目标：统一 `main.py` 中的启动入口和初始化顺序，避免未来加逻辑时出现双入口维护偏差。

- [ ] 对齐广告子仓库默认配置
  目标：清理 `telegram_monitor_ads_dev` 和 `ads_repo` 中仍然存在的占位配置，减少默认值与主仓库表现不一致的问题。

## 处理原则

- 优先修稳定性和真实用户会踩到的问题。
- 不为了“更优雅”做大范围重构。
- 保留广告模块的商业硬依赖，不削弱现有收入模型。
